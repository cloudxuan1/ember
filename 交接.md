## 项目概览

- 域名：`ember.cloudxuan1.com`
- 服务：FastAPI，跑在 VPS Docker Compose，公网走 Cloudflare Tunnel

**名称对照**：`ember` = 本项目（也是 VPS 主机名）｜`memory` = 旧系统（Haven Fork，跑在 `/root/memory`，域名 `memory.cloudxuan1.com`，只作参考样板，不拆不修不迁）｜`chat-lite` = 独立的自建聊天前端项目，与本仓库无关。

## MCP

- 端点：`https://ember.cloudxuan1.com/<MCP_PATH>/`（streamable HTTP，有状态 SSE 模式——claude.ai 只认这种）
- 鉴权三层（`backend/app/oauth.py`）：随机路径 → 授权页口令 → 每个请求验 Bearer token。密钥位置见「仪表盘 → 常用命令」，不写明文。
- 客户端接入：
  - **Claude Code**（✅ 已验收）：`claude mcp add --scope user --transport http ember "<完整URL>/" --header "Authorization: Bearer <token>"`
  - **claude.ai 网页 connector**（✅ 已接通，2026-07-09）：走标准 OAuth 流程；接通过程见「历史日志」及本节「经验教训」
  - **ChatGPT**：理论兼容（标准 OAuth 2.1），未实测
- 五个工具：`memory_search` / `memory_recall` / `memory_save` / `memory_list` / `memory_status`，实现在 `backend/app/mcp_server.py`；第六个工具 `memory_briefing(topic?)` 是主动浮现入口。**工具的 docstring 就是发给客户端模型的说明书**，改行为必须同步改说明。
- **space 语义**：search/list 不传 space = 只搜 personal 核心层；项目话题显式传（ember/vps/...）；跨全库传 `"all"`。入口统一在 `memories.resolve_space()`。
- `换 embedding 模型` 操作步骤见「仪表盘 → 常用命令」。

## 审核台

- 地址：`https://ember.cloudxuan1.com/review`（手机浏览器直接用）；登录态 30 天。
- 草稿入口：`POST /review/api/drafts`，与 MCP 的 token 刻意分开（持 MCP token 的一方不该开得了审核台）；单条或批量 `{"drafts": [...]}`，**整批原子**。
- 流程：提取推草稿（pending，带 batch 批次名）→ 手机上 通过/改/删 → 通过才写入 memories + sources。
- **反悔区**：右上"↩ 已审核"，通过/删过的都能撤回重审；通过的撤回会连生成的记忆一起删；被 `superseded_by` 引用的拒绝撤回会 409。
- 实现：`backend/app/drafts.py`（存取）+ `backend/app/review.py`（页面+API）。

## 经验教训

- FastMCP 默认开 DNS-rebinding 防护且 Host 白名单只有本机，放在反代/Tunnel 后面公网请求会被 421 拒——要显式传 `TransportSecuritySettings` 把公网域名加进 `allowed_hosts`。
- Claude MCP client 不跟 307 redirect，遇到无尾斜杠路径会误判成 sign-in 失败。根因是 Starlette Router 默认 `redirect_slashes=True` 触发 307。修法：包一层 `_MCPPathFix` ASGI 中间件原地改写路径。
- **claude.ai 网页 connector 排障六层洋葱**（2026-07-03 起，2026-07-09 剥完接通）：① 其客户端不跟 307 → 中间件改写路径；② 只认有状态 SSE，无状态 JSON 判失败；③ 现行 connect 流程强制走 OAuth；④ `token_type` 必须大写 `"Bearer"`（其后端 model 是 `Literal["Bearer"]`，小写静默炸）；⑤ token 交换后一度看似"静默中止"；⑥ **真凶：Cloudflare「Block AI training bots」**——claude.ai 的 OAuth 请求（python-httpx UA，Azure IP）与换 token 后的正式 MCP 请求（UA `Claude-User`，出口段 `160.79.104.0/21`）来源不同，后者被 CF 边缘静默丢弃。关掉该开关（Overview 页下拉框，与 Bot Fight Mode 是两个开关）即通。同款案例：claude-ai-mcp #438、#125、#519。排障方法论：拿旧系统当对照组逐项消差异；给请求头/请求体加透视日志（脱敏）拿实锤，别靠猜；"排除一个功能 ≠ 排除整个层"——查过 Bot Fight Mode 没开就以为 CF 无责，漏了独立的另一个开关。
- V5 换 embedding 模型是常规操作，流程见「仪表盘 → 常用命令」，中间态安全降级不会炸库。
- 模块级懒状态要给"独立进程第一个入口"留路：`_vec_status` 只在 `get_conn()` 里点亮，`python -m app.embeddings` 单独跑时没人开过连接就来问会误报——教训通用：任何"初始化时顺手设置"的全局状态，都要想想不走常规初始化路径的入口。
- docker-compose 的 `environment` 是显式白名单：加新环境变量必须同步加透传行，否则容器里静默拿不到（V3/V5 两次踩同一坑）。
- claude.ai 删 connector 后立刻重加同 URL 会报 "already exists"（幽灵记录）→ 等几分钟或末尾加斜杠再试。
- CC auto 模式分类器会拦「AI 自改权限/配置」「生产库裸 SQL 写」「无条件 DELETE」这类操作——遇到停下来让轩决定，设计时尽量走应用层 API 绕开。
- 导入功能"暂停不能继续"是旧系统最痛的坑——ember 的导入管线为此专门设计断点续跑和人工质检。
