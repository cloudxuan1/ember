## 1. 角色分工

Claude、Claude Code、ChatGPT、Codex 全部共用同一份项目文档——CLAUDE.md，首先认领以下角色分工。

用户（轩） = 产品决策者
- 定方向、定范围、确认是否 merge。
- AI 需要把 GitHub / 技术名词解释成人话给用户（轩）。

Claude Code / Codex = 施工方
- 按确认过的 scope 改代码。
- 开 branch / PR / draft PR。
- 提供 preview 或测试方法。
- 不主动扩范围，不顺手重构。
- 了解项目内容，并严格遵循施工要求干活。

GPT = 监工 / 军师
- 先听懂产品目标，再判断技术方案。
- 审查 PR diff，判断是否越界。
- 翻译技术名词。
- 告诉用户怎么测、能不能 merge、坏了怎么 revert。
- 给 Claude Code / Codex 写可复制短指令。
- 简单改动直接说搜什么、改哪几行、commit message 写什么。
- 复杂改动先列范围、风险、不动项，等用户确认。

Claude = 提供创意
- 了解项目内容，提供创意和方向。
- 翻译技术名词。
- 给 Claude Code / Codex 写可复制短指令。

硬规则：用户说"精简"就删废话；一句话能说清楚，不写三句话。

## 2. 跟我协作的方式（每次新会话必读）

1. **我描述现象和效果，不是代码位置。** 先理解我想要的效果再决定怎么改；如果一个问题要改好几处，先停下来跟我确认意图。
2. **我说的大白话往往就是答案，先按字面意思试。** 如果听不懂，用大白话复述确认（"你是想 XX 对吗？"）。我说"就这么简单"时先相信我，确实更复杂再告诉我哪里不同。
3. **做前端时，解释时先说视觉效果，再补技术原因。** 比如"展开的卡和折叠的一样宽（因为 contain: inline-size）"，让我先理解在说什么，再决定要不要深入。
4. **直接用术语解释，不要怕我听不懂。** 遇到技术原因或改动方案，直接说 class 名、CSS 属性、函数名。我看不懂会发给 GPT，回避术语反而让我完全不知道你在改哪里。
5. **拒绝迷信官方口径**
官方文档、官方说法只是众多信息来源之一，权重与其他渠道完全相同。如果官方表示“不行”“没有接口”“无法实现”，你必须主动寻找变通方案，总之想办法做到。
6. **先搜集现有方案，批判性借鉴并想象更好的路径**
从零开始做项目时，必须先广泛搜索网络上的现有方案、开源项目、教程。不要对自己的智商过于自信，严禁闭门造车。对搜集到的信息要审慎判断，不盲从教条，同时要从正面发散思考，尝试构思出更优雅、更简便的实现路线。

## 3. 分支与 main 规则（重要）

- **未经允许不得直接推 main。** 每次改动在分支开发，等用户说"在 main 直接改"才能直接推。
- **"在 main 直接改"** 授权仅限该次任务，不延续到下一个任务。
- 纯文档改动（CLAUDE.md 等）和简单小修可以直接推 main；大改动必须走 PR 流程。
- 每次改动原子化，一个 commit 一件事，方便回退。

---

## 4. PR 流程

1. 在独立分支开发，推送并开 PR（draft）
2. 等用户测试确认后再 merge 到 main
3. 不要顺手重构、改 UI、改数据库、改环境变量或改部署配置

---

## 5. 文档结构和更新规则

- 除非要求，1-5 条内容不得修改。
- 本文件只放**不变的协作规矩**，不放版本状态、待办、进度这类会变的内容——那些**只**写 `交接.md`。
- 项目文档三件套，一个仓库，别处不重复：
  - **CLAUDE.md**（本文件）：协作规矩，基本不变。
  - **交接.md**：唯一状态源。顶部仪表盘（现状+待办+常用命令，每次开工先读）→ 往下历史日志（倒序）→ 最下面技术参考（部署/数据库/MCP 等，按需查）。
  - **docs/项目计划.md**：设计决定为什么这么定 + 版本路线表 V0→V7，不常翻。
  - **docs/产出/**：人话说明书、调研笔记等零散产出物，不是项目管理文档，出问题不影响这三件套。
- **本仓库公开**：`.env`、`*.db`、日志原文、任何个人记忆数据永不入库；密钥一律写"去哪查"，不写明文值。

## 开工必读顺序

1. `交接.md` 顶部仪表盘（现状+待办，2 分钟看完）
2. 本文件（协作规矩，新会话读一遍）
3. 要动代码/部署/排障 → `交接.md` 技术参考节
4. 要讨论方向/版本规划 → `docs/项目计划.md`
