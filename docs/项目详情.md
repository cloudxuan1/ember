## 架构

```text
V1 当前链路：
客户端 / 浏览器
  → Cloudflare Tunnel（ember 专用 tunnel 容器，与旧系统的 ombre-tunnel 各自独立）
  → 同一 compose 网络内 http://ember:8000
  → FastAPI /health
（宿主机 127.0.0.1:8300 → 容器 8000 只作本机调试，同旧系统 18001 的用法）

长期链路（V2+ 逐版加）：
客户端（Claude.ai / ChatGPT / Claude Code / Codex / 自写）
  → MCP / REST
  → FastAPI
  → SQLite ember.db（data/ 卷，后期 sqlite-vec 向量表）
```

## 技术栈和配置

- Python 3.12 + FastAPI + uvicorn（版本见 `backend/requirements.txt`）
- 存储：SQLite 单文件（V2 起），放 `data/` 卷 —— 备份 = `cp`，库是可从日志原文重建的派生索引
- 配置：`.env`（gitignore，样例见 `.env.example`）
- 端口约定：宿主机 `127.0.0.1:8300` → 容器 `8000`，只绑回环，公网只走 Tunnel

## 部署和运维

- VPS：主机名 ember，2G 内存 + 2G swap，已跑旧系统 memory（占用 `127.0.0.1:18001`，ember 用 8300 避开）。**连接方式、IP、Tunnel 账号等见本地《VPS维护文档》（不入库）。**
- 部署步骤：
  1. Cloudflare Zero Trust → Networks → Tunnels → **新建**名为 `ember` 的 tunnel，拿到 token（不碰旧系统 memory 的 tunnel）
  2. 给新 tunnel 加 public hostname：`ember.cloudxuan1.com` → `http://ember:8000`（tunnel 容器和 ember 容器在同一 compose 网络，用服务名直连）
  3. VPS 上 `git clone` 本仓库到 `/root/ember`（与旧系统 `/root/memory` 平行，互不影响）
  4. `cp .env.example .env`，填入 `TUNNEL_TOKEN`，**把 `MCP_PATH` 改成随机长串**（`echo "mcp-$(openssl rand -hex 16)"`），**并生成三个 OAuth 值**（`EMBER_OAUTH_PASSWORD` / `ACCESS_TOKEN` / `REFRESH_TOKEN`，各 `openssl rand -hex 32`）——OAuth 留空 = 门禁关闭 = 记忆接口只剩路径一层保护
  5. `cd /root/ember && docker compose up -d --build`
  6. `curl http://127.0.0.1:8300/health` 验证本机可达，再访问 `https://ember.cloudxuan1.com/health` 验收
- 更新：`git pull && docker compose up -d --build`
- 回滚：`git checkout <上一个 tag/commit> && docker compose up -d --build`
- 备份（V2 起）：每日 `cp data/ember.db data/ember.db.bak` + 每周压缩包 + 重大改动前手动快照

## 数据库

V2 起 SQLite（`data/ember.db`，Docker 卷挂载）。五张表：`memories` / `memory_edges` / `memory_sources` / `memory_drafts` / `memory_embeddings`（V5 语义指纹，普通表非 vec0 虚拟表，每行带 model 列，记忆删除时级联删），结构见 `docs/施工计划.md` 第 2 节，建表代码在 `backend/app/db.py`（两处必须保持一致）。

## MCP

- 端点：`https://ember.cloudxuan1.com/<MCP_PATH>/`（streamable HTTP，**有状态 SSE 模式**——claude.ai 只认这种，无状态 JSON 模式下 initialize 200 也会被判"连不上"；末尾斜杠可有可无，服务端自动处理）
- **MCP_PATH / OAuth / embedding 密钥在哪找**：`ssh ember 'grep -E "MCP_PATH|EMBER_OAUTH|EMBER_REVIEW|EMBEDDING" /root/ember/.env'`（不写进任何公开文档/仓库）
- **鉴权三层**（实现在 `backend/app/oauth.py`）：随机路径 → 浏览器授权页口令（`EMBER_OAUTH_PASSWORD`）→ 每个请求验 Bearer token。OAuth 环境变量留空 = 门禁关闭且发现端点全部 404（仅限本地开发）
- **客户端接入**：
  - **Claude Code**（✅ 已验收）：`claude mcp add --scope user --transport http ember "<完整URL>/" --header "Authorization: Bearer <token>"`——支持自定义请求头，不走 OAuth 流程
  - **claude.ai 网页 connector**：现行 connect 流程强制走 OAuth（well-known 404 直接判死；官方文档"可选"的说法过时），当前被上游 bug 挡住——anthropics/claude-ai-mcp#519：token 交换成功后其后端静默中止。服务器侧已用官方 MCP Python SDK 客户端完整验证（注册→授权→token→initialize→tools/list 全过），等上游修复后重试即可
  - **ChatGPT**：门禁为标准 OAuth 2.1，理论兼容（旧系统有先例），未实测
- 五个工具：`memory_search` / `memory_recall` / `memory_save` / `memory_list` / `memory_status`，实现在 `backend/app/mcp_server.py`；**工具的 docstring 就是发给客户端模型的说明书**，改行为必须同步改说明
- **space 语义（V6 隔离）**：search/list 不传 space = 只搜 personal 核心层；项目话题显式传（ember / vps / ...）；跨全库传 `"all"`。入口统一在 `memories.resolve_space()`，关键词腿/语义腿/list 统计同一规则
- `main.py` 的 `_WireLog` 是**临时排查组件**（记录 MCP/OAuth 请求头 + register/token 请求体，已脱敏），claude.ai 接通验收后移除

## 审核台（V3）

- 地址：`https://ember.cloudxuan1.com/review`（手机浏览器直接用）；登录口令 = `EMBER_OAUTH_PASSWORD`，登录态 30 天（签名 cookie，密钥 = `EMBER_REVIEW_TOKEN`，重启不失效）
- 草稿入口：`POST /review/api/drafts`，带 `Authorization: Bearer <EMBER_REVIEW_TOKEN>`——**与 MCP 的 token 刻意分开**（PR #6 评审 P1：持 MCP token 的一方含 claude.ai 后端不该开得了审核台）；单条对象或 `{"drafts": [...]}` 批量，**整批原子**（坏一条整批不写，脚本可放心重试）
- 流程：提取会话推草稿（pending，带 batch 批次名）→ 手机上 通过 / 改 / 删 → 通过才写入 memories + memory_sources；拒绝的行保留 rejected 状态，`memory_drafts` 表兼任导入审计记录
- **反悔区**：页面右上"↩ 已审核"进入；通过/删过的都能撤回 pending 重审，通过的撤回会连生成的记忆（含来源、边）一起删；被 superseded_by 引用的拒绝撤回（409）
- tags 支持中文逗号/顿号，入库自动归一成英文逗号
- 门禁关闭（本地开发）时审核台免登录，与 MCP 行为一致；登录口令连错 5 次锁 60 秒
- 实现：`backend/app/drafts.py`（存取）+ `backend/app/review.py`（页面 + API）

## 经验教训

- 旧系统（Haven Fork）证明了 VPS + Docker Compose + Cloudflare Tunnel + MCP 这条线上链路可行，但代码耦合重，小改动牵连大逻辑——所以 ember 从干净仓库开始，每版只做一条真实可跑的小链路。
- 导入功能"暂停不能继续、脱水不带 identity"是旧系统最痛的坑——ember 的导入管线为此专门设计断点续跑和人工质检（见施工计划）。
- FastMCP 默认开 DNS-rebinding 防护且 Host 白名单只有本机，放在反代/Tunnel 后面公网请求会被 421 拒——要显式传 `TransportSecuritySettings` 把公网域名加进 `allowed_hosts`（V2 部署时踩到）。
- **claude.ai 网页 connector 排障五层洋葱**（2026-07-03，每层都有日志实锤）：① 其 MCP 客户端不跟 307 重定向 → 进程内改写路径；② 只认有状态 SSE 传输，无状态 JSON 下 initialize 200 也判失败；③ 现行 connect 流程必须走 OAuth，well-known 404 直接判死（旧系统 memory 是老流程时代连上的存量户，不能当"无鉴权可行"的证据）；④ token 响应 `token_type` 必须大写 `"Bearer"`——其后端用 MCP Python SDK（UA python-httpx），模型是 `Literal["Bearer"]`，小写静默炸；⑤ 以上全修后仍卡在其后端 token 交换后静默中止——上游 bug（anthropics/claude-ai-mcp#519），已用官方 SDK 客户端全流程反证服务器无责。
- 排障方法论两条：拿同一 Cloudflare zone 上的旧系统当**对照组**逐项消差异；给服务器加请求头/请求体透视日志（脱敏）拿照片级证据，别靠猜。教训：Bot Fight Mode 从未开启却被当嫌疑人半晚——**先查配置再定罪**。
- token 响应带 `scope` 字段与能连上 claude.ai 的服务商（WorkOS/Entra 等）对齐；缺 scope 是否为 #519 触发条件待其修复后验证。
- Claude MCP client 不跟 307 redirect，遇到 307 会转而探测 OAuth 端点，最终报 sign-in 失败（表象很误导）。根因：Starlette Router 默认 `redirect_slashes=True`，请求 `/<MCP_PATH>`（无尾斜杠）被 307 跳到 `/<MCP_PATH>/`。修法：在 FastAPI app 外包 `_MCPPathFix` ASGI 中间件，在路由前原地把路径改写成带斜杠版本，客户端直接拿到 200（V2 修复时踩到）。
- **V5 两个反直觉取舍**：① FTS5 默认分词器不切中文（整句一个词），关键词腿沿用 LIKE 子串匹配反而对中文好使，千条量级性能无虞；② sqlite-vec 的 vec0 虚拟表把向量维度写死进 DDL，换 embedding 模型要拆表——改用普通表存 float32 BLOB + `vec_distance_cosine()` 暴力扫描，维度只是数据，毫秒级。
- 模块级懒状态要给"独立进程第一个入口"留路：`_vec_status` 只在 `get_conn()` 里点亮，`python -m app.embeddings` 单独跑时没人开过连接就来问 → 误报扩展不可用（codex 审出的 P1）。修法：`vec_available()` 惰性探测，未探测先开一次连接。教训通用：**任何"初始化时顺手设置"的全局状态，都要想想不走常规初始化路径的入口**。
- docker-compose 的 `environment` 是显式白名单：加新环境变量必须同步加透传行，否则容器里静默拿不到（V3 的 `EMBER_REVIEW_TOKEN`、V5 的 `EMBEDDING_*` 两次都是同一坑）。
