# ember 施工计划（定稿版）

> 由 v0.3 融合计划定稿：底盘决定不变，融入一个实际运行系统的记忆模块经验（导入管线、软失败、审核机制），版本编号按现状重排（V0 = 调研，已完成）。
> 一句话：**目标要深，地基要稳，施工要分层，回滚要清楚——但魂不能丢。ember 首先是窗口之间不灭的那点火，然后才是好用的记忆库。**

---

## 0. 定位

- 核心：**关系的连续性**。窗口关闭不等于遗忘。
- 项目记忆（VPS / 部署决策 / 协作状态）是有用的附加，不是主体。
- relationship 不是十种 type 里的一种，它是这个库的底色；spaces 用来保护核心层不被项目记忆稀释。

## 1. 底盘决定（不再重议）

| 维度 | 决定 | 理由 |
|---|---|---|
| 存储 | **SQLite** 单文件（后期 sqlite-vec） | 千条量级、单用户；`cp` 即备份；没有 `down -v` 删库这回事 |
| 部署 | Docker Compose + Cloudflare Tunnel | 复用已跑通环境，减少变量；不上 Nginx/certbot/supergateway |
| 证据 | memory_sources **轻引用**指向日志 | 原文本体留在私有日志仓库；库是可重建的派生索引，库丢了 ≠ 数据丢了 |
| 字段 | 砍 importance/confidence/heat/activation_count | 服务于已否决的"权重自动浮现"，是死字段；用 tier 离散分层代替 |
| 类型 | 用 tags，不立 type 枚举 | 与 tags 功能重叠 |
| 导入 | **提前到 V2 试点、V3 全量** | 带两个月存量进场，导入是立项主因 |
| 浮现 | 语义共振主动检索 + 冷却 + sensitive 不碰 | 不做热度代谢自动浮现 |

## 2. 数据模型

三个维度：`tier`（稳定性）/ `tags`（类型主题，自由）/ `space`（领域，personal 是核心底色）。

```sql
CREATE TABLE memories (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    date        TEXT NOT NULL,           -- 事件真实日期（从日志抠，不是导入日）
    content     TEXT NOT NULL,           -- 原话 + 一句上下文，永不被摘要覆盖
    tags        TEXT DEFAULT '',         -- 逗号分隔，可检索
    tier        TEXT DEFAULT 'normal',   -- anchor / normal / process（锚点只在这里标，不设重复的 is_anchor 位）
    topic       TEXT DEFAULT '',         -- 主题/实体，去重合并的钩子
    space       TEXT DEFAULT 'personal', -- personal(核心) / ember / vps / ...
    start_date  TEXT,                    -- 区间型起点（点事件留空）
    end_date    TEXT,                    -- 区间型终点；状态不存死，读时用今天现算
    is_resolved INTEGER DEFAULT 0,
    superseded_by INTEGER REFERENCES memories(id),
    created_at  TEXT DEFAULT (datetime('now','+8 hours'))
);
CREATE INDEX idx_mem_date  ON memories(date);
CREATE INDEX idx_mem_tier  ON memories(tier);
CREATE INDEX idx_mem_topic ON memories(topic);
CREATE INDEX idx_mem_space ON memories(space);

CREATE TABLE memory_edges (
    id       INTEGER PRIMARY KEY AUTOINCREMENT,
    from_id  INTEGER NOT NULL REFERENCES memories(id),
    to_id    INTEGER NOT NULL REFERENCES memories(id),
    relation TEXT DEFAULT 'related'  -- led_to / same_as / contradicts / supersedes / related
);

CREATE TABLE memory_sources (
    memory_id  INTEGER NOT NULL REFERENCES memories(id),
    source_ref TEXT,   -- 指向私有日志仓库的哪份文件哪一段（轻引用，不存全文）
    quote      TEXT    -- 关键原话片段
);
```

原则：相同的记忆（同一事被多个窗口各提一次）→ 合并成一条主记忆；相近但独立的两条 → **绝不合并**，两条 + 一条 `led_to` 边。年轮（重读感受层）放 V6+。

## 3. MCP 工具（V2 先做五个）

```text
memory_search(query, space?, limit=8)   返回目录（短结果 + ID），不塞全文
memory_recall(id)                        取一条完整详情 + 来源证据
memory_save(date, content, tags, tier, topic, space)
memory_list(space?, date_range?, page)   强制分页 + 先给统计概览
memory_status()                          服务/库状态：活着吗、几条、最近一次写入（调试连接必备）
```

纪律写进工具本身：列目录永远分页；search 只返目录；真正 recall 才算"被使用"。不用裸 exec 类工具，记忆操作一律走专用工具。

**架构提醒**：MCP 模式下模型自己伸手调工具，没有"每轮开口前"的后端钩子。真正的主动浮现和哨兵预判要等远期自建网关；近期打折版 = briefing 工具 + connector 说明里写"提到过去的事先 memory_search"。

## 4. 导入管线设计（借鉴一个实际运行系统的实战经验）

两个月存量日志（claude.ai 导出 JSON 为主 + 手动维护的 .md 总结为辅）是本项目主要资产，管线要求：

1. **锚点逐组推进**：日志按组解析，每组提取的记忆全部成功写入后才更新进度锚点；中断重跑不漏不重（断点续跑——旧系统"暂停不能继续"之痛的解药）。
2. **分组格式带来源**：喂给提取模型的每行带 `[时间戳][id=来源标识] 说话人: 内容`，让模型能引用具体来源行 → `memory_sources.quote` 自动有着落。
3. **草稿 → 人工审 → 入库**：提取结果先进草稿，人工接受/编辑/拒绝后才进 memories；拒绝的不落库。（参考系统的注释写着"以前自动直接应用，吃过亏后改成先出草稿"。）
4. **来源校验防幻觉**：模型引用的来源 id 若在日志里不存在，整条丢弃。
5. **审计日志**：每批导入/去重合并写一行审计（old_ids → new_ids），这是"所有操作可回滚"的具体机制。
6. **软失败**：单条解析失败记录跳过，不崩整个批次。

## 5. 版本路线（V0 = 调研，空壳从 V1 起）

| 版本 | 内容 | 验收 |
|---|---|---|
| **V0 调研** ✅ | 五系统对比、旧系统试用、两轮计划 | 已完成（2026-06） |
| **V1 空壳上线** | FastAPI `/health` + Docker + Tunnel + 文档 | `ember.cloudxuan1.com/health` 通；`/root/memory` 不受影响 |
| **V2 记忆链路 + 试导入** | SQLite 建表 + 五个 MCP 工具 + 导入脚本试切一份真实日志 + 人工质检 | claude.ai 连上能 save/search/recall；一份日志正确导入（时间戳对、切错处人工拆对） |
| **V3 存量全导入 + 去重 + 证据** | 全量日志导入（含断点续跑）、去重合并、memory_sources、审计日志 | 千条级正确入库；相近不同的两条各自独立且有边；抽查去重对 |
| **V4 关系边 + 区间时间** | memory_edges 完善 + start/end 读时现算 | led_to 边在；区间状态随"今天"自动变 |
| **V5 语义检索** | sqlite-vec + hybrid（FTS5/向量 + RRF）；embedding 写入时生成、API 挂了存 NULL 不崩、rebuild_embeddings() 事后补、维度不硬编码 | 搜"难过"能召回同义不同词的表达 |
| **V6 空间隔离 + 主动浮现** | space 检索隔离；浮现打折版（三级漏斗：未解决 → 话题相关 → 近期，上限 8 条；语义共振 + 冷却 + sensitive 不碰）；可选 Digest 只出建议 | — |
| **V7 迁旧 + 退役** | `/root/memory` 有价值数据迁入 → 测试 space 抽样核对 → 稳定后切主域名 → 旧系统留观察期 | 旧记忆可搜；关键锚点不丢；可回滚 |

每版独立 branch + 小 PR，merge 前监工审 diff；一个 PR 不同时改库 + MCP + 前端 + 部署。

## 6. 远期（不进主线，避免稀释重心）

- Digest / Dream：只出建议（重复合并 / 冲突提示 / 升 anchor / 可 resolved 候选），人工确认才动。
- 自建 API 网关 + 分层 prompt 缓存（稳定画像 / 近期摘要 / 会话摘要 / 动态召回目录）；哨兵式预判（一次便宜模型调用判断要不要搜/搜什么）到这一层才可移植。
- 年轮（memory_comments）：旧记忆挂重读时的新感受，原话一字不动。
- L4 SOP 程序记忆、recall_logs 活跃信号。
