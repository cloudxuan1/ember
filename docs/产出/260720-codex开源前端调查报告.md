# 开源聊天前端选型调查：给 ember 自建前端“抄作业”

> 调查日期：2026-07-20  
> 面向：产品负责人（轩）  
> 调查范围：只评估前端产品形态、接入成本、群聊能力、魔改空间、许可证和项目生命力；不涉及代码实现。  
> 注：GitHub 星标数是调查当日附近的页面快照，只用于判断社区量级，不代表产品质量。许可证判断不是法律意见，正式商用前仍应让专业人士复核。

## 一句话结论

**正式版 ember 不建议整体 fork 任何一家。推荐“自写一个很薄的手机壳 + 定点抄各家成熟交互”。**

原因不是这些项目不好，而是 ember 的目标并不是“又一个 ChatGPT 网页版”：它要把聊天、群聊、日记/读书/音乐/纪念日历/健康数据、全部设置和记忆审核台装进同一个手机式三栏产品。现有候选大多已经自带账号、数据库、聊天记录、记忆、RAG、Agent、权限和后台；整体拿来后，开发工作会从“做 ember 前端”变成“拆掉别人半个产品，再把 ember 塞进去”。

明确回答三个最重要的问题：

- **正式方案：自写壳，不整体采用。**聊天页可参考成熟项目的组件和交互，但 ember 自己掌握底部三栏、路由、主题、账号状态和记忆审核入口。
- **如果只为尽快做功能原型：Open WebUI 最快。**它能直接接 OpenAI 兼容接口，PWA 完整，而且已有 Channels：同一时间线里可以先 `@GPT`、再 `@Claude`，共享上下文。但它不适合作为最终换皮底座，原因是过重、后端重复、品牌许可证受限。
- **“手机三栏 app 感”最接近的是 LobeHub（原 LobeChat）的移动端方向。**它有独立移动入口、PWA/移动适配、原生移动应用、主题系统和 Agent Group；但它仍不是 ember 要的“聊天 / 小手机桌面 / 设置”三个固定底栏，且当前主仓库非常重、许可证不适合无顾虑地做商业换皮。

## 先把两个容易混淆的词说清楚

本报告把“多个模型”拆成三种完全不同的东西：

1. **并排比较**：同一个问题同时发给 GPT、Claude，左右或上下展示两份答案。Open WebUI 的 Multi-Model、big-AGI 的 Beam、LibreChat 的 Multi-Convo 主要属于这一类。
2. **后台 Agent 协作**：一个主 Agent 在后台叫其他 Agent 做事，最后只把汇总结果交给用户。LibreChat 的 Subagents/Agent Chain 更接近这一类。
3. **真正的群聊时间线**：用户、GPT、Claude 都作为“群成员”出现在同一个聊天流里，能指定谁说、让下一位接着说，甚至由主持模型决定轮到谁。**这才是轩描述的“用户当群主”。**目前最有参考价值的是 Open WebUI Channels、LobeHub Agent Group 和 ChatClaw Team Chat。

因此，“支持多模型”并不自动等于“支持 ember 群聊”。

## 总览表

等级含义：强＝基本现成；中＝能做但要改；弱＝需要重做。这里评的是“对 ember 的适合度”，不是项目绝对好坏。

| 项目 | 体量/拆壳 | 手机/PWA | 接 ember 接口 | 真群聊 | 整体换脸 | 许可证安心度 | 活跃度 | 对 ember 的角色 |
|---|---|---:|---:|---:|---:|---:|---:|---|
| **LobeHub（原 LobeChat）** | 很重，主仓库难拆 | 强 | 中-强 | **强** | 技术强、许可受限 | 中-弱 | 很强 | 最接近愿景的产品样板，不建议整仓 fork |
| **Open WebUI** | 很重，前后端一体 | **强** | **强** | **强（Channels）** | 技术中、许可受限 | 中-弱 | **极强** | 最快功能原型；抄群聊与设置台 |
| **LibreChat** | 很重，企业聊天平台 | 中-强 | **强** | 中（协作强、群聊形态弱） | 中 | **强（MIT）** | 很强 | 抄断线续传、分支、模型切换和管理能力 |

## 逐家评估

## 1. LobeHub（原 LobeChat）

项目：[GitHub](https://github.com/lobehub/lobehub) · [许可证](https://github.com/lobehub/lobehub/blob/main/LICENSE) · [移动端下载页](https://lobehub.com/downloads) · [Agent Group RFC](https://github.com/lobehub/lobehub/discussions/8920)

### 1）技术栈与体量

- 当前已经从“漂亮的 LobeChat”长成完整的 LobeHub Agent 平台。主仓库是大型 monorepo（一个仓库里包含许多应用和内部包），核心是 React、Next.js、Vite、Ant Design、Drizzle ORM，并带桌面端、数据库、鉴权、知识库、插件、Agent、任务、渠道等大量能力。
- 仓库的依赖清单非常长，并有专门的 web、mobile、desktop 构建入口。它不是“拿走聊天组件就结束”的小壳。
- **拆壳判断：难。**直接 fork 再删除功能，长期跟上游同步会持续冲突。更合理的做法是研究其交互，或单独评估它公开的 `@lobehub/chat` UI 包，而不是搬整个产品。

### 2）手机端 / PWA 成色

- 强。官方明确提供 PWA、移动适配，当前构建里还有独立 mobile SPA 入口；下载页已经提供 iOS 和 Android 应用。
- 它是本次候选中最重视“像产品、不是像后台管理网页”的一档，动效、弹层、会话模式、触控布局都值得看。
- 但“移动完成度高”不等于“已经是 ember 三栏”：它的导航仍围绕 Agent 工作台，不是聊天/趣味桌面/设置三个固定底栏。

### 3）接 ember 的 OpenAI 兼容后端

- 能接。官方长期支持自定义 OpenAI 代理地址和多供应商。
- 现实难点不在“请求能不能通”，而在它自己的模型网关、会话、Agent、记忆、文件、数据库逻辑会与 ember V7 重叠。只把 `base URL` 指向 ember 很容易；让所有高级功能正确服从 ember 的会话和记忆语义则要做适配。

### 4）多模型群聊 / 多 Agent 同场

- 强，而且比普通“并排比较”更接近目标。Agent Group 的设计包含：从现有 Agent 组成群、主持模型判断下一位发言者、公开消息/私信、顺序或自然发言、最大连续回复数、群系统提示等。
- 这套设计非常值得 ember 抄：**群成员列表 + 主持人规则 + 谁下一位发言 + 防止无限互聊 + 群内私信是否可见。**
- 当前仓库和 issue 已有正式的 group-chat 功能痕迹，不再只是早期概念；但自托管版本具体功能边界仍应在真正采用前做一次部署验收。

### 5）主题魔改自由度

- 技术上强：官方一直强调自定义主题，UI 组件体系也完整。
- 但主产品的视觉语言非常鲜明，而且布局、组件、状态管理、Agent 业务已经深度结合。要“完全不留 Lobe 味”，工作量不会小。
- 更大的问题是许可证：换脸后若作为商业衍生品分发，可能必须取得商业许可。

### 6）许可证

- 当前主仓库不是普通 MIT/Apache，而是 **LobeHub Community License**：基于 Apache 2.0，但增加条件。
- 原样商业使用可以；如果“基于 LobeChat 开发并分发衍生作品”，许可证正文写明需要商业许可。
- 对个人自用通常无碍；对 ember 将来公开、收费、给多人部署的路线不够干净。不要被旧文章里“Apache 2.0”或 `package.json` 的旧字段误导，应以仓库根目录当前 LICENSE 为准。

### 7）社区活跃度

- 极强。调查时约 8 万星、1.5 万 fork，版本和 PR 更新非常密集。
- 反面是变化快：今天借来的内部结构，明天可能又重构。它更适合持续观察设计，不适合作为 ember 必须紧跟的上游母体。

### 8）抄什么

- 移动端弹层、输入框、消息气泡/文档两种阅读模式。
- Agent 头像、身份卡、群成员管理。
- 主持模型选下一位发言者、自然/顺序发言、连续回复上限。
- 主题 token（统一控制颜色、圆角、间距、阴影）的思路。
- **不要抄整套账号、数据库、记忆和 Agent 网关。**这些正是 ember 已经有或需要自己掌控的部分。

**结论：最像 ember 愿景的“样板间”，但不是最合适的地基。**

## 2. Open WebUI

项目：[GitHub](https://github.com/open-webui/open-webui) · [PWA 文档](https://docs.openwebui.com/getting-started/open-webui-as-app/) · [OpenAI 兼容接入](https://docs.openwebui.com/getting-started/quick-start/connect-a-provider/starting-with-openai-compatible/) · [Channels](https://docs.openwebui.com/features/channels/) · [许可证说明](https://docs.openwebui.com/license/)

### 1）技术栈与体量

- 前端是 SvelteKit/Svelte 5 + Tailwind，后端是 Python/FastAPI，内含账号、权限、数据库、RAG、文件、语音、图片、工具、模型管理、分析后台等。
- 它是“完整 AI 平台”，不是纯前端。即使 ember 后端也是 FastAPI，也不能因此认为两者容易合并；它们是两套各自拥有数据模型和业务规则的 FastAPI。
- **拆壳判断：很难。**把聊天组件从大量 Svelte store、路由、Open WebUI API 和权限逻辑里抽出来，成本可能高于重写薄壳。

### 2）手机端 / PWA 成色

- 强。官方写明每个实例都是 PWA，可安装到手机主屏、全屏运行，并支持通知；更新记录也持续处理 iOS 图标、旋转和侧滑等移动细节。
- 手机上属于“成熟的响应式聊天平台”，但整体气质仍偏工具/管理平台，不像轻巧的个人手机桌面。

### 3）接 ember 的 OpenAI 兼容后端

- 本次最容易的一档。后台 Connections 可以直接填 OpenAI 兼容 URL 和 API Key；支持 `/models` 自动发现，也可手填模型 ID。
- ember 若提供标准的 `/v1/models`、`/v1/chat/completions` 和 Bearer 鉴权，接入最顺；若只实现了聊天接口，没有模型列表，也能用白名单手工配置。
- 需要确认 ember 的流式输出、工具调用、reasoning 字段和附件是否完全符合它预期；“基础聊天能通”不代表所有按钮都能用。

### 4）多模型群聊 / 多 Agent 同场

- 现在有两套不同能力：
  - Multi-Model Chat：同一问题同时问多个模型，响应并排或在手机上堆叠，之后还可 Merge（合并答案）。
  - **Channels：真正的共享时间线。**多个真人和多个模型处于同一频道，先 `@gpt` 起草，再 `@claude` 反驳；消息、线程、表情、置顶、文件和上下文都留在同一条时间线。
- Channels 与 ember“用户当群主”很接近，但默认是**点名谁，谁回答**，不是 LobeHub 那种主持模型自动决定轮到谁。ember 可以同时抄两者：手动 `@成员` 必须有，自动主持作为可选模式。

### 5）主题魔改自由度

- 技术上可改，Svelte + Tailwind 并不封闭；但完整界面组件数量庞大，换掉整体信息架构仍是大工程。
- 官方把完整 white-label（白标，即去掉原品牌换成自己的品牌）和高级主题列入企业授权。当前许可下，大规模部署不能随意移除或弱化 Open WebUI 品牌。

### 6）许可证

- v0.6.6 以后使用自定义 Open WebUI License，不是 OSI 认证的标准开源许可证。
- 关键限制：默认必须保留清晰的 Open WebUI 品牌；滚动 30 天内不超过 50 个最终用户、获得书面许可或购买企业许可时，才有例外。
- 个人/小规模自用比较宽松；若 ember 以后公开推广并完全换脸，这是明显风险。旧的 v0.6.5 仍是 BSD-3，但停在旧版本意味着放弃后续安全修复和功能。

### 7）社区活跃度

- 本次最强量级。调查页面约 14.6 万星、2.1 万 fork，提交、发布、文档和安全公告都非常活跃。
- 也意味着产品越来越大，未来会继续朝“AI 平台”而不是“可嵌入聊天壳”发展。

### 8）抄什么

- **Channels 的群聊时间线**：`@模型`、线程、表情、置顶、未读、附件和共享上下文。
- 手机 PWA 安装引导、断点网络下的反馈、侧滑手势。
- 模型选择器、模型头像、能力标签和管理员控制哪些模型可见。
- 设置后台的信息分组；可借鉴它如何把大量参数分层，但不要把全部高级选项直接扔给普通用户。
- 多模型并排比较和 Merge，可作为 ember 群聊之外的“对答案”模式。

**结论：功能原型第一名；最终换皮底座不推荐。**

## 3. LibreChat

项目：[GitHub](https://github.com/danny-avila/LibreChat) · [自定义 OpenAI 兼容端点](https://www.librechat.ai/docs/quick_start/custom_endpoints) · [兼容矩阵](https://www.librechat.ai/docs/compatibility) · [Subagents](https://www.librechat.ai/docs/features/subagents) · [许可证](https://github.com/danny-avila/LibreChat/blob/main/LICENSE)

### 1）技术栈与体量

- React 18 + Vite + TypeScript，后端是 Node/Express 系，数据侧通常需要 MongoDB；完整 Docker 方案还可带 MeiliSearch、RAG API 等。前端依赖包含 Radix、Tailwind、Monaco、React Query、文件和 Artifact 渲染。
- 这是企业级、多用户、多供应商 AI 门户。账号、权限、Agent、MCP、搜索、文件、审计都做得很深。
- **拆壳判断：难。**它比 Open WebUI 更接近 React 技术生态，但仍不是纯前端组件库。只拿聊天壳会碰到大量 LibreChat 自有 API 和数据结构。

### 2）手机端 / PWA 成色

- 中上。前端明确使用 PWA 插件，社区也确实把它安装到手机；响应式聊天可用。
- 但官方文档没有像 Open WebUI 那样把“手机 App 体验”当一等产品叙事，历史上也出现过 Android 返回键、新会话状态等移动问题。
- 更像“手机上能很好使用的桌面聊天平台”，不是为底部三栏生活 App 从零设计。

### 3）接 ember 的 OpenAI 兼容后端

- 强。`librechat.yaml` 可定义自定义端点、base URL、模型列表、鉴权、额外参数和自定义 Header；官方兼容矩阵明确支持 OpenAI-compatible streaming。
- 它的配置能力比 NextChat 更精细，遇到 ember 的自定义 Header、模型别名、reasoning 字段时更容易调。
- 代价是要引入 LibreChat 自己的服务器、配置和数据层，只为转发到 ember 显得绕。

### 4）多模型群聊 / 多 Agent 同场

- Multi-Convo 可以做多响应流/并排比较；Agent Chain 和 Subagents 能做多 Agent 后台协作、串联或委派。
- 但这些都不天然等于“GPT、Claude 各有头像，在同一群聊时间线里自然轮流说话”。要做轩要的群聊，仍需新做参与者、发言路由和时间线 UI。
- 它适合抄“Agent 怎么协作”，不适合直接抄“群聊长什么样”。

### 5）主题魔改自由度

- React + Tailwind/Radix，技术上可彻底改；MIT 也允许换品牌。
- 但现有 UI 明显是 ChatGPT/企业工作台思路，组件和功能面太广。整体换成 ember 的手机桌面仍会触及路由、侧栏、账号和大量弹窗。

### 6）许可证

- **MIT，商用和修改都友好。**保留版权和许可声明即可。
- 在本次大型平台候选里，它的法律路径最省心。

### 7）社区活跃度

- 很强。调查时约 4.1 万星、8.4 千 fork，近期版本仍持续加入 Subagents、Skills、管理台、断线恢复和安全增强。
- 更新速度快，配置和部署也相对复杂，需要跟 changelog 处理破坏性变化。

### 8）抄什么

- **Resumable Streams**：手机切后台、网络断开、换设备后，生成中的答案还能续上。这对“手机为主”非常值钱。
- 中途切模型但保留上下文、Fork 对话分支、编辑重发。
- Agent Builder、模型参数分层、管理员控制普通用户能看到什么。
- 附件、Artifact 侧栏和长对话整理。

**结论：可靠的企业能力参考库；对 ember 来说仍然过重。**

## 4. NextChat（原 ChatGPT-Next-Web）

项目：[GitHub](https://github.com/ChatGPTNextWeb/NextChat) · [许可证](https://github.com/ChatGPTNextWeb/NextChat/blob/main/LICENSE) · [PWA manifest](https://github.com/ChatGPTNextWeb/NextChat/blob/main/public/site.webmanifest)

### 1）技术栈与体量

- React 18 + Next.js 14 + Zustand + Sass，另有 Tauri 桌面壳；依赖和结构明显比 LobeHub、Open WebUI、LibreChat 轻。
- 官方主打首屏约 100 KB、桌面包约 5 MB、浏览器本地存储。虽不是今天最现代的架构，但“把聊天壳拆出来”是本次最容易的一档。
- **拆壳判断：较容易。**会话列表、消息流、输入框、设置和本地存储边界相对直观。

### 2）手机端 / PWA 成色

- 强。官方明确支持响应式、暗色和 PWA，manifest 使用 standalone；产品一直以跨平台轻客户端为卖点。
- 体验偏“轻量 ChatGPT 客户端”，加载快，适合手机；但没有 ember 的三栏生活 App 导航，也没有丰富的群成员交互。

### 3）接 ember 的 OpenAI 兼容后端

- 强。`BASE_URL` 可以覆盖 OpenAI 地址，也支持自部署模型和自定义模型列表。
- 基础聊天大概率是最少阻力之一。仍应验证它在 URL 路径、模型发现、工具调用和新 reasoning 格式上的假设。

### 4）多模型群聊 / 多 Agent 同场

- 弱。能选不同模型、用 Mask（预设角色/提示词），但核心仍是一条会话对应一次模型响应。
- 没有现成的群成员、`@模型`、主持人路由或多 Agent 共享时间线。群聊基本要另做。

### 5）主题魔改自由度

- 强。项目较小、Sass/CSS 较直接、品牌约束少，换布局和视觉比大型平台轻松。
- 但如果加入三栏、趣味桌面、审核台和群聊，最终会改掉大量页面；届时“fork NextChat”与“用其交互作参考后自写壳”的差距已经不大。

### 6）许可证

- **MIT，商用友好。**

### 7）社区活跃度

- 星标很高（约 8.8 万），但不能只看星数。当前主包仍是 Next.js 14/React 18，最近正式 release 页面显示的稳定版时间明显慢于 LobeHub/Open WebUI/LibreChat；GitHub 组织页虽在 2026 年仍有更新，整体节奏属于中等而非前沿高速。
- 适合借成熟轻壳，不适合指望上游替 ember 做群聊和生活模块。

### 8）抄什么

- 首屏速度、本地优先、简单 PWA 安装。
- 手机会话列表、输入框、暗色模式、对话导入导出。
- Mask/角色模板的轻量交互。
- 自动压缩长对话的用户提示方式。

**结论：最好的“轻聊天壳参考”，但不是群聊底座。**

## 5. big-AGI

项目：[GitHub](https://github.com/enricoros/big-AGI) · [文档](https://big-agi.com/docs) · [Beam / 多模型说明](https://big-agi.com/openai) · [许可证](https://github.com/enricoros/big-AGI/blob/main/LICENSE)

### 1）技术栈与体量

- Next.js 15 + React 18 + MUI Joy/Emotion + Zustand + Dexie/IndexedDB，并有 Prisma、tRPC 和可选数据库能力。
- 比 LobeHub、Open WebUI 少一层“企业全家桶”，但它自身功能密度很高：多供应商、Beam、语音、图片、文件、Persona、Inspector、同步等。
- **拆壳判断：中等偏难。**代码强调模块化，但产品核心就是高性能多模型工作台，不是极简聊天组件。

### 2）手机端 / PWA 成色

- 中上。官方长期提供可安装移动 App/PWA，并在 2.0 做过 Mobile UI overhaul（移动 UI 大改）、侧栏 peeking 等。
- 手机上功能很多，适合高级用户；对 ember 的普通生活型入口来说，信息密度可能过高。

### 3）接 ember 的 OpenAI 兼容后端

- 强。官方文档写明任意 OpenAI-compatible endpoint 可直接工作，并可自动发现模型/能力。
- 客户端优先、API Key 和会话可留在浏览器，做单人自用很方便；如果 ember 负责统一账号和数据，则要避免双重本地状态。

### 4）多模型群聊 / 多 Agent 同场

- Beam 是本次“多模型比较与融合”最强的参考：多个模型并行回答，再用 Fusion 合并、交叉检查或综合。
- 但它仍不是人物化群聊。模型更像多个并行专家窗口，不是同一微信群里的成员。要做 GPT 接 Claude 上一句继续说，仍需新路由。

### 5）主题魔改自由度

- MUI Joy/Emotion 有集中主题系统，MIT 允许自由换脸；技术上中上。
- 现有界面带有强烈的“专家工作台”个性，完全变成生活 App 需要重新组织导航和功能密度。

### 6）许可证

- **MIT，商用友好。**

### 7）社区活跃度

- 约 7 千星、1.6 千 fork，量级不如前三，但维护非常积极；调查时 activity 页面几天内仍有连续提交。
- 核心维护更集中，项目巴士系于少数人，是比大社区更高的长期风险。

### 8）抄什么

- **Beam：一次问多个模型、展示差异、选中优秀答案、合并答案。**
- AI Inspector：把实际发出的模型、参数、token、费用透明展示给高级用户。
- Persona 切换和每模型参数，但应藏在二级设置，不要污染普通聊天页。
- 本地优先、快速打开和多面板在手机上的收纳方式。

**结论：把它当“多模型实验室”抄，不要把它当群聊成品。**

## 6. Chatbot UI（mckaywrigley）

项目：[GitHub](https://github.com/mckaywrigley/chatbot-ui) · [依赖清单](https://github.com/mckaywrigley/chatbot-ui/blob/main/package.json)

### 1）技术栈与体量

- Next.js 14 + React 18 + Tailwind + Radix，后端数据强依赖 Supabase（PostgreSQL、鉴权、存储）。
- UI 本身不算巨型，但“启动一个聊天前端”要先带起 Supabase，这与已有 FastAPI ember 后端不搭。
- **拆壳判断：中等。**组件可以读，但要去掉 Supabase 数据流并重新接 ember，已经接近改写。

### 2）手机端 / PWA 成色

- 有 `next-pwa`，可以安装；基本响应式可用。
- 官方 README 反而写着“更好的 mobile layouts 正在做”，说明现版移动体验不是它最有把握的部分。

### 3）接 ember 的 OpenAI 兼容后端

- 中弱。它支持 OpenAI、Anthropic、Gemini、Ollama 等，但官方当前 README 没有像 Open WebUI/LibreChat/big-AGI 那样给出成熟的任意 OpenAI-compatible base URL 配置路径，并把“better backend compatibility”列为后续更新。
- 改代码当然能接，但那就失去“直接采用”的意义。

### 4）多模型群聊 / 多 Agent 同场

- 弱。支持多个供应商、Assistant/Tool 等，不等于同场群聊；没有现成群成员时间线和主持机制。

### 5）主题魔改自由度

- 技术与许可都自由。Tailwind/Radix 很适合换视觉。
- 但这项优势被维护停滞和 Supabase 耦合抵消。

### 6）许可证

- GitHub 标记为 **MIT**，商用友好。

### 7）社区活跃度

- 约 3.3 万星，但健康度弱。第三方仓库元数据显示主分支已约一年未实质更新，GitHub 仍积压大量 issue/PR；README 的“大更新即将回来”也长期没有落地。
- **高星不等于现在适合开新项目。**

### 8）抄什么

- 基础消息组件、文件选择、Prompt/Assistant 管理的简单做法。
- 不建议抄它的整体数据架构，也不建议等它的“大更新”。

**结论：历史参考，不进入最终候选。**

## 7. ChatClaw（额外补充，专看真群聊）

项目：[GitHub](https://github.com/fastclaw-ai/chatclaw) · [许可证](https://github.com/fastclaw-ai/chatclaw/blob/main/LICENSE)

### 1）技术栈与体量

- Next.js 16 + TypeScript + Tailwind/shadcn；默认 Dexie/IndexedDB，也可用 Drizzle + SQLite；通过 SSE 接 OpenAI 兼容网关。
- 相比大型平台轻得多，产品核心就是“公司—Agent—团队—聊天”，因此群聊相关代码和界面更集中。
- **拆壳判断：较容易。**但它也带 OpenClaw 配置文件、Agent workspace 等特定概念，不能无脑搬。

### 2）手机端 / PWA 成色

- 官方 README 没有明确承诺 PWA，也没有找到像 Open WebUI/LobeHub 那样的移动专项文档。
- 可视为响应式 Web 候选，但“手机主力、可安装、离线壳、刘海/底部安全区、后台恢复”目前**缺少官方证据**，不能当成熟能力采购。

### 3）接 ember 的 OpenAI 兼容后端

- 强。官方明确写明可接 OpenClaw gateway 或任意 OpenAI-compatible API，并使用 SSE 流式输出。
- 不过它的 Agent/team 行为原本建立在 OpenClaw 概念上；接 ember 时要确认普通 OpenAI 兼容模式是否覆盖团队成员配置、会话 key 和文件能力。

### 4）多模型群聊 / 多 Agent 同场

- **非常强，语义最直白。**可与单个 Agent 私聊，也可与 Team 群聊；群内 Agent 顺序回复，每个 Agent 看完整上下文；消息像 Discord/Telegram，可有多段流式消息。
- 这与“GPT、Claude 都接进来，用户当群主”高度贴合。缺的是成熟度，不是方向。

### 5）主题魔改自由度

- Tailwind/shadcn + MIT，换脸自由；头像还内置随机、Emoji、上传三种模式。
- 项目规模小，重做底部三栏比改大型平台轻松。

### 6）许可证

- **MIT，商用友好。**注意这里只说 ChatClaw 前端仓库；不要把它与 FastClaw 后端的社区许可证混为一谈。

### 7）社区活跃度

- 非常新，调查时只有约 200 多星、三十多个 fork、极少 issue。不能用“大量真实用户验证过”来背书。
- 适合当设计样本或短期实验，不适合把 ember 的长期命运绑在它上面。

### 8）抄什么

- 团队/群成员选择器、每个 Agent 的头像和身份。
- 群内顺序发言、完整共享上下文、多段消息流。
- 单聊与群聊共用一套会话列表。
- 创建群、编辑成员、删除确认、默认 Agent 防误删。

**结论：群聊交互参考第一名之一；成熟底座最后一名之一。**

## 哪家最接近“手机底部三栏 App”

**严格答案：没有一家。**所有候选都把“聊天/Agent 工作台”当整个产品，而 ember 的中栏是一个生活功能桌面，右栏还要吞下所有设置与记忆审核。这已经不是给聊天网页加两个按钮，而是全新的一级信息架构。

如果必须排“接近程度”：

1. **LobeHub：整体感觉最接近。**移动端投入最高，已有 Agent Group，主题与交互精致；最像未来成品的气质。
2. **Open WebUI：功能结构第二接近。**PWA + Channels + 设置/管理台已经齐，但视觉更像强大的工作后台。
3. **ChatClaw：群聊语义最接近。**团队、成员、头像、顺序发言非常对题，但移动/PWA和社区成熟度不够。
4. **NextChat：手机轻壳最接近。**快、简单、可安装，但群聊和中栏桌面都没有。

所以可以用一句容易记的话概括：

> **看 LobeHub 学“成品感”，看 Open WebUI 学“群频道和后台”，看 ChatClaw 学“AI 群成员”，看 NextChat 学“轻和快”。**

## 最终推荐方案

## A. 正式版：自写薄壳 + 抄各家交互（推荐）

这里的“自写”不是从零发明聊天技术，而是只自己掌握 ember 独有的外壳和状态边界：

- 一级导航固定为底部三栏：聊天 / 趣味桌面 / 设置。
- 聊天页只负责显示、输入、群成员和调用 ember V7；模型、记忆注入和最终会话语义由 ember 后端负责。
- 趣味桌面从第一天就是独立一级页面，不必塞进别人“插件市场”或“工具侧栏”。
- 设置与记忆审核台共用 ember 自己的权限和数据，不再绕过第三方后台。
- 主题从一开始就用统一设计变量管理，后续换颜色、圆角、字体、阴影和密度时不会逐页返工。

这条路的主要收益：

- 不背负第二套账号、聊天数据库、记忆系统和权限后台。
- 不受 LobeHub/Open WebUI 的换牌限制。
- 群聊可按轩的产品定义设计，而不是硬套“并排回答”。
- 中栏未来加日记、音乐、纪念日历、健康数据时，不需要和上游聊天项目打架。

## B. 如果要先做一个一两周可见的功能验证

可临时部署 **Open WebUI** 对接 ember，重点验证两件事：

1. ember 的 OpenAI 兼容流、模型列表、认证和记忆注入能否被成熟客户端稳定消费。
2. 用 Channels 模拟“一个群里先叫 GPT、再叫 Claude”的体验，观察用户到底更喜欢手动 `@成员`，还是自动主持轮流说。

这个原型的定位必须写清：**验证 API 和群聊流程，不是未来前端代码底座。**否则原型一旦好用，很容易被误当成最终产品，后面反而难拆。

## 群聊功能建议：直接综合三家，而不是选一家

ember 的群聊第一版可以用以下产品规则，分别来自已经验证过的开源思路：

- 默认手动点名：像 Open WebUI Channels，用户输入 `@GPT` 或点头像，避免每句话都烧多个模型费用。
- 可选“全员回答”：像 big-AGI/Open WebUI Multi-Model，同一问题让选中的成员都答，手机上按时间线依次显示，不必强行分栏。
- 可选“自然讨论”：像 LobeHub，由一个便宜的主持模型决定下一位发言者。
- 顺序模式：像 ChatClaw，按群成员顺序回复，逻辑简单、行为可预期。
- 必须有停止条件：最多几位回复、最多连续几轮、费用预估、随时“停止讨论”。否则 AI 会互相客套到停不下来。
- 每条消息清楚显示：成员头像、模型名、耗时、是否用了记忆、是否调用工具。普通用户默认只看头像和名字，高级信息点开再看。

## 各家值得抄的最终清单

| ember 页面/能力 | 主要抄谁 | 抄的具体内容 |
|---|---|---|
| 手机整体质感 | LobeHub | 动效、弹层、触控密度、移动会话切换、主题变量 |
| 群聊主时间线 | Open WebUI Channels | `@模型`、线程、表情、置顶、未读、共享上下文 |
| 自动主持讨论 | LobeHub Agent Group | 下一位发言者、自然/顺序、连续回复上限、公开/私信 |
| 群成员和团队 | ChatClaw | 成员头像、身份、建群、团队顺序回复、单聊/群聊切换 |
| 多模型对答案 | big-AGI Beam | 并行回答、选择、融合、交叉检查 |
| 手机断网/切后台 | LibreChat | 可恢复流式响应、跨标签/设备续接 |
| 轻量与首屏 | NextChat | 本地优先、少层级、快速启动、简单 PWA |
| 高级设置/管理 | Open WebUI + LibreChat | 参数分组、模型可见性、权限和高级项渐进展开 |
| 记忆审核台 | **ember 自己** | 不抄第三方记忆后台，直接围绕 ember 的真实记忆结构设计 |
| 趣味功能桌面 | **普通手机桌面范式** | 图标网格、小组件、最近使用；现有聊天前端没有合适答案 |

## 不建议做的几件事

- 不要因为某项目星标高，就把它当“未来一定有人维护”。Chatbot UI 是最明显反例。
- 不要把“同时问多个模型”包装成“群聊”；两者的情感体验、费用和控制方式完全不同。
- 不要先 fork LobeHub/Open WebUI 再计划“以后慢慢删”。删除别人完整平台的耦合，通常比重写薄壳更慢。
- 不要让第三方前端再保存一份权威记忆。ember 才是记忆库，前端只能展示、建议和审核。
- 不要第一版就让所有 Agent 自动无限互聊。先做手动 `@` 和严格轮数上限，费用和行为都更可控。
- 不要把全部温度、top-p、上下文长度、工具开关直接堆在普通设置首页。设置页要分“常用 / 模型高级 / 记忆审核 / 系统”。

## 推荐决策

建议轩现在拍板为：

> **ember 前端采用“自主手机 PWA 薄壳”路线；不 fork 完整聊天平台。LobeHub 作为视觉与自动群聊标杆，Open WebUI Channels 作为群时间线标杆，ChatClaw 作为成员/团队交互样本，NextChat 作为轻量实现参照，big-AGI Beam 作为以后可加的多模型对答案模式。**

如果必须选一个现成项目做临时验证，选 Open WebUI；如果只问最终成品气质最像谁，选 LobeHub；如果只问 AI 群聊动作最像谁，选 Open WebUI Channels + ChatClaw；如果问正式产品底座，答案仍然是 **ember 自己的薄壳**。
