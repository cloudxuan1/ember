# kiwi-mem 人话拆解笔记（给轩的学习材料）

> 2026-07-09 Fable 写。照你学 aion 的路子：先看全貌，再跟一条消息走一遍，然后逐个机制拆。
> 每个机制都标了「在哪个文件」和「咱抄不抄、为什么」。读完 kiwi 就进你的素材库了。
> 版本：仓库 e35a354（2026-07 初），AGPL 协议——**只借思想，一行代码不能抄**（抄了 ember 就得跟着开源同协议）。

## 〇、一句话定位

kiwi-mem 和 ember 网关是**同一类东西**：都是站在"你的聊天界面"和"AI 服务商"中间的转发网关，路过时把记忆塞进去、把新记忆捞出来。区别在血统：

- **kiwi**：产品化的通用件，谁都能装，所以什么都做成了开关和配置（面板 200 多个配置项），自己不带聊天窗（用户拿现成客户端接它）
- **ember**：你的私人件，记忆的"审核台守门"哲学是它独有的——kiwi 提取完**直接入库**，ember 提取完进**草稿区等你批**。这是两家最大的三观差异，后面反复会碰到

她的技术栈：FastAPI + PostgreSQL（向量用 pgvector 插件）。咱们：FastAPI + SQLite（向量用 sqlite-vec）。思路同源，她是"正经数据库"，咱是"一个文件就是全部家当"——单用户够用且好备份。

## 一、一条消息在 kiwi 体内的旅程（main.py，5300 行的主干）

你发"我昨天把星露谷的压缩系统抄完了"，kiwi 依次干这些：

1. **组装带记忆的 system prompt**（`build_system_prompt_with_memories`，main.py 709 行起）。注入顺序是精心排的，后面第二节细讲
2. **转发给主模型**，流式回传给你（支持 OpenAI 和 Anthropic 两种格式）
3. **回复路过时做安检**：检查末尾有没有模型藏的 `<!--emotion:高-->` 情绪标记、`<!--dream:trigger-->` 做梦标记，撕下来（用户看不见），存干净的正文
4. **后台异步提取记忆**（`process_memories_background`，1158 行起）：对话先存库；攒够 N 轮（可配，默认 5）触发提取——注意这一切在后台跑，聊天体验零延迟
5. 提取出来的记忆**直接入库**，顺手做去重和矛盾检测

对照 ember 网关：1、2、4 咱们都有（哨兵→briefing+search→组 prompt→主模型→存库→回流），3 没做（情绪标记，见第五节），5 咱们故意不一样（进草稿区不直接入库）。

## 二、prompt 注入顺序——"省 90% 钱"的那个排法

这是 kiwi 最值钱的工程经验之一。API 有个机制叫 **prompt 缓存**：如果这次请求的开头和上次一模一样，服务商只收 1/10 的钱。所以排列原则是——**越不变的越往前放**：

```
── 静态区（一天内不变，命中缓存）──
人设 → 用户画像 → 日历摘要 → 项目指令
── 动态区（每轮都变，放最后，反正也缓存不了）──
当前时间 → 语义搜索到的记忆 → 梦境场景 → 犯困提示
```

有个细节见功力：**当前时间必须放动态区**。她在注释里写了血泪教训（issue #8）——时间每分钟变，放在静态区里等于从那一行起整段缓存全部作废。

**咱抄了吗**：抄了，ember 网关的 `build_messages` 就是稳定块在前、记忆和历史在后。但咱们没做她那种多断点精细缓存（她插了 CACHE_B1/B2 好几个缓存分界点）——单用户聊天量小，省的钱不够复杂度的利息，这是昨天调研时就拍板跳过的。

## 三、热度系统——"会淡忘"是怎么算出来的（database.py `calculate_heat`）

kiwi 每条记忆有个 0~1 的"热度"，公式是人话版的艾宾浩斯遗忘曲线：

- **初始温度**由重要度和情绪浓度决定（越重要/越动情，起点越热）
- **随时间衰减**，用半衰期模型（普通记忆 3 天减半，重要记忆 7 天）
- **被聊到就升温续命**：每次被召回，半衰期延长 50%——反复聊到的事越记越牢
- **锁定记忆永远 1.0**：你亲手标重要的，永不降温

热度决定待遇（这招很漂亮）：**高热度全文注入，中热度只给个模糊摘要（后面还标注"（印象模糊）"），低热度不打扰**。AI 对旧事真的会说"我记得有这么回事，细节想不起来了"——不是装的，是它拿到的就是模糊版。

还有个容易漏看的好设计：**冷启动保护**——从没被搜到过的新记忆不走遗忘曲线，按重要度给个保底热度。不然新记忆没人聊就直接冻死了。

**咱抄吗**：**现在不抄，将来的遗忘机制拿它当蓝图**（调研时已拍板）。原因：热度要靠 access_count（被召回次数）喂数据，ember 聊天页还没上线、记忆还没开始"被使用"，现在做是无米之炊。等网关跑起来积累了召回数据，做遗忘时直接照这个公式打样。

## 四、矛盾检测——"朋友生了"怎么自动翻页"朋友怀孕"（database.py `detect_contradictions`）

咱们已经抄进 reflow 的那招，看看她的原版多省钱——**全程不调 API，纯本地算**：

1. 新旧记忆**标题**像不像（字符重叠度 > 0.7 → 算同一主题）
2. **内容**相似度在 0.4~0.85 之间 → 判"相关但说法变了"=疑似矛盾。**大于 0.85 不算矛盾算重复**（去重管），这个区间设计很精
3. 命中就把旧记忆标失效 + 自动连一条 supersedes 边

还有两条保护规则值得记：**手动录入的记忆和锁定记忆永远不被自动标失效**——机器只能翻机器写的页,人写的只有人能动。

**咱抄的版本**：ember 走的是"LLM 提取时顺便建议 supersedes 边 → 进草稿 → 你审核确认"。她是机器全自动翻页，咱是机器建议、你盖章——又是那个三观差异。她的纯计算版胜在便宜可靠，咱们的胜在有人把关，各得其所。

## 五、情绪标记——让主模型顺手当情绪传感器（main.py 729 行 + 1078 行）

很妙的白嫖设计：在 system prompt 里嘱咐主模型"这轮对话情绪浓时，在回复最末尾藏个 `<!--emotion:高-->`"。回复路过网关时撕下标记，情绪浓度存进记忆的 emotional_weight——**没多调一次模型，情绪检测白送**。另有关键词规则引擎兜底（"抱抱""崩溃""太好了"这类词直接判）。

情绪浓度接进热度公式：动情的记忆衰减更慢——人也是这样，触动过你的事忘得慢。

**咱抄吗**：调研时标了"值得借"，但依赖方（热度系统）还没做，所以**等做遗忘机制时一起上**。到时候网关加两行：注入嘱咐、撕标记存字段，成本极低。

## 六、提取防重——已经抄到手的那招（memory_extractor.py）

她的提取 prompt 里有一段《已知信息处理【最重要】》：把库里相关的旧记忆列给提取模型看，明说"相同的忽略、补充的可提、矛盾的标更新"。配合入库前的相似度硬去重，双保险。

她喂"已知信息"的取材是**搜索相关 50 条 + 最近 30 条**合并去重——比只看"最近的"聪明，早年的种子记忆也能被对比到。

另有一个朴素但有效的**META_BLACKLIST 硬过滤**：提取结果里带"记忆库""检索""bug""端口"这些词的直接扔——防止你们讨论记忆系统本身的话被当成记忆存进去（不靠模型自觉，硬过滤）。

**咱抄的版本**：`reflow.py` 的 `_known_memories()` 就是这招（已在分支上，有测试）。META 黑名单咱没做——因为有审核台，垃圾提取你一眼就删了；她没有守门人，只能靠黑名单。同一个问题，闸门硬的人可以少写代码。

## 七、Dream 做梦 + 日历压缩——她的"记忆整理"双件套（dream.py / daily_digest.py）

- **日历**：每天 0 点把当天碎片合并成"日摘要"，再逐级卷成周/月/季/年。注入时近的给全文、远的给概括——去年的事只记个轮廓
- **Dream**：手动或闲时触发，干三件事：清过时碎片、把相关碎片融合成"记忆场景"（带叙事和因果）、推断你没明说的事（"前瞻"）。prompt 写得像小说："你刚刚睡着了，最近的记忆碎片开始浮现……"

**咱抄吗**：**都不抄**（调研时拍板）。这两样解决的是"记忆多到看不过来"的规模问题——她面对的是无人审核、自动堆积的库，必须靠机器整理。ember 每条记忆都过你的手、你的审核本身就是最高质量的"整理"，79 条的库离"需要做梦"还远。但 Dream 的"碎片融合成场景"这个想法先记着，等你的库上千条时可能真香。

## 八、工具抽屉——按需拿工具，别全摆桌上（tool_drawer.py）

kiwi 内置 20 多个工具，全塞给模型每轮白烧几百 token。她的解法：给工具分类打标签+关键词，**你这句话像是要用哪类工具，就只把那几个拿出来**，其余收抽屉里。

**咱抄吗**：不用抄——ember 的 MCP 才 6 个工具，全摆桌上也不占地方。但**这个思路值得记**：它本质是"按需注入、不浪费上下文"，跟咱们哨兵判断"这句要不要搜记忆"是同一个哲学的不同应用。

## 九、跟你的前端方案直接相关的两样（admin-panel/）

这部分建议你讨论前端方案前扫一眼，因为正对你的需求：

1. **config-schema.js——配置项登记表**。她把 200 多个配置做成一张"权威登记表"：每项登记中文标签、类型、默认值、控件形态、**人话说明**（她注释原话："解决『忘了这参数干嘛』"）。页面照表自动渲染，加配置=登记一行，不用写页面代码。**你要的"模型/接口/key 页面可填"，照这个模式做就是正解**——ember 配置比她少得多，一张小表就够
2. **她的面板是裸 HTML+JS，没上 React/Vue 这些框架**。单人用的管理界面，这个选择省掉整个前端工程链（不用 build、不用 npm、改完刷新就生效）。ember 审核台已经是这个路数，聊天页可以继续——这也是"轻壳"的题中之义

## 十、总结账本

| 机制 | 咱的态度 | 状态 |
|------|---------|------|
| 静态前置吃缓存 | ✅ 抄了 | 网关分支上 |
| 提取时注入已知记忆防重 | ✅ 抄了 | reflow.py，有测试 |
| 矛盾检测→supersedes 边 | ✅ 抄了（改成"建议+你审"） | reflow.py，有测试 |
| 情绪标记 | 🕐 等遗忘机制一起上 | 记在这 |
| 热度衰减/淡忘 | 🕐 将来遗忘机制的蓝图 | 记在这 |
| 配置登记表（前端） | 👀 前端方案讨论的输入 | 你扫一眼第九节 |
| Dream / 日历五层压缩 | ❌ 不抄，规模不匹配 | — |
| 工具抽屉 / 多供应商路由 / 多断点缓存 | ❌ 不抄，单用户不值 | — |

最后说个读完的整体感受：kiwi 全部聪明才智都花在"**没有人看着时，机器怎么不搞砸**"——去重、黑名单、矛盾检测、自动整理，全是给"无人审核"打的补丁。ember 选了另一条路：**有个人守着闸门，机器只需要勤快，不需要绝对正确**。两条路没有高下，但你选的这条有个副产品——每条记忆你都见过，所以你永远不会对自己的记忆库感到陌生。这就是你说的"不喜欢黑箱"的制度化。
