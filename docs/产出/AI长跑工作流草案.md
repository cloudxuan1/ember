# AI 长跑工作流草案（给轩的讨论底稿）

> 2026-07-08 深夜 Fable 写。轩想学"让 AI 干十小时"的工作流；她自己说出了关键：
> **"前提在于我们完全聊好了方案和计划，我才能放心放手。"** ——这就是全部秘密，本文只是把它展开。

## 一句话原理

长跑 = 把"人盯着"换成"机器盯着"。AI 干一段 → 自动验收（测试/构建）→ 红了自己修、绿了进下一项 → 循环。
**长跑的质量 = 验收闸门的质量**，闸门软的活放手就是灾难。

## 咱们已经有的闸门（家底不错）

| 闸门 | 现状 |
|------|------|
| 自动测试 | ember 111 个，一秒跑完，全 mock 不花钱 ✅ |
| 审核台 | 记忆入库的人工闸，AI 绕不过你 ✅ |
| 分支 + PR | 代码不碰 main，随时可扔 ✅ |
| 部署快照 | 每次部署前拍库备份，可回滚 ✅ |
| 交接文档 | 断了随时接（本仓库交接.md + 冲刺计划）✅ |

## 什么活能放手，什么不能

- **能放手**：边界清楚 + 判分客观的活。例：补测试、按定稿设计施工、批量提取日志草稿（反正有审核台兜底）、修一列明确的 bug
- **不能放手**：要品味和产品判断的活。例：前端长什么样、功能取舍、prompt 的语气——这些跑十小时只会跑偏十小时
- **灰色地带**：放手但设检查点——干 N 步自动停，等你看一眼再续（遛狗模式，咱们今天就这么干的）

## 技术上怎么启动（CC 现成能力，不用装东西）

1. **`/loop` 命令**：让 CC 按节奏反复执行一个任务，自己定间隔
2. **后台子代理**：派 Sonnet 干活，干完自动回报（今天精读 kiwi-mem 就是这么干的）
3. **定时任务（schedule/cron）**：定点自动跑，比如"每晚 3 点提取昨天的聊天回流"
4. 组合：主循环规划 + 子代理执行 + 测试当闸门 + 卡住了留言等你

## 放手前置清单（聊方案时逐项打勾，勾不满不放手）

- [ ] 目标一句话说得清（做完长什么样）
- [ ] 验收标准机器能判（测试怎么算过、哪些文件该出现）
- [ ] 边界写死（哪些文件/系统不许碰）
- [ ] 失败预案（卡住怎么办：重试几次？留言暂停？）
- [ ] 检查点节奏（每多少步/小时停下汇报）
- [ ] 回滚路径（干砸了怎么退）

## 明天讨论的问题（顺序建议）

1. **前端方案**（轩说不草率）：从"每天怎么用它"聊起 → 形态定了技术就顺了
2. **新工作流程**：轩说不再 Codex 审核——新流程什么样？（谁审 PR / 要不要审 / 派活规则）
3. **第一次长跑试点选什么活**：建议选个闸门硬的（比如"前端方案定稿后的施工"就很合适——设计聊死，施工放手）
